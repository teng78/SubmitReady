## 一、Codex 的主要贡献与暴露的问题

本项目中，Codex 最直接的贡献是把篇幅很长的课程要求转化为可以落实和验证的模块边界。SubmitReady 最终形成了安全解压、规则管理、确定性检查、受控命令运行、报告与历史记录五个核心模块，并补充了 React 前端、FastAPI 后端、SQLite 持久化、Docker、CI 和文档。Codex 尤其适合处理机械但容易遗漏的安全边界，例如把 Zip Slip、盘符路径、UNC 路径、符号链接、重复成员、加密成员、文件数量和解压体积限制统一到安全解压服务中。

但这次实践也说明，AI 生成的模块不会天然兼容。最典型的问题是后端 `/api/rules` 返回嵌套的 `rule` 对象，报告使用 `rule_name` 和 `return_code`；前端 Mock 却假设规则已经扁平化，并使用 `report.rule` 和 `exit_code`。前后端各自的测试都通过，真实上传流程仍然失败。这让我认识到，测试数量不等于测试质量。如果 Mock 是根据开发者的想象编写的，它可能只是重复了同一个错误假设。最终修复方法是让前端测试使用真实后端响应结构，并通过浏览器完整执行“上传—报告—历史”路径。

第二个问题来自冷启动。主工作区已经安装依赖，因此 lint、类型检查和构建都能通过；在全新导出的副本中，pnpm 11 却拒绝执行 esbuild 的安装脚本，第一次冷启动失败。项目把策略修改为只允许 `esbuild: true`，重新导出另一个干净副本后，才完成 21 个后端测试、6 个前端测试和生产构建。这个过程让我意识到，分发问题经常来自锁文件、权限策略、缓存和隐式环境，而不只是业务代码。README 中的命令必须由干净环境验证，不能因为“理论上可行”就写成已经通过。

## 二、规约、计划与测试纪律

规约阶段最有价值的动作是独立上下文冷读。审阅者只读取 SPEC 和 PLAN，没有此前对话，就指出 YAML 字段、默认值、危险命令策略、规则权威源和 ZIP 跨平台路径语义仍然含糊。修订之后，SafeExtractor 才有了稳定错误码和明确清理职责，规则层也明确了字段范围、重复导入和命令白名单。这说明 SPEC 的价值不在于篇幅，而在于陌生实现者能否不依赖隐含上下文做出一致实现。

当前开发环境没有提供课程指定的 Superpowers `test-driven-development` 等技能，因此本项目不能声称完整遵循了技能强制的红—绿—重构流程。后端保留了首次 `ModuleNotFoundError: app` 的失败记录以及最终 21 tests green；前端也经历了 lint 和真实 API 契约问题，但部分实现与测试在同一子任务中快速形成，不等同于逐测试的严格 TDD。这是本项目过程证据中的真实缺口。另一方面，自动化门禁仍然发挥了约束作用：接口修复后必须重新通过 ESLint、TypeScript、Vitest 和 Vite build；依赖策略修复后必须在第二个全新目录重跑，而不能只修改文档。

## 三、为什么它不是 Agent

SubmitReady 的检查顺序、命令来源和停止条件均由普通代码预先确定。用户点击检查后，系统不会让 LLM 决定下一步、选择工具或根据结果修改代码。模板解释 Provider 只接收已经产生、截断并脱敏的错误摘要，执行一次输入和一次输出。即使删除整个解释模块，安全解压、规则、检查、Runner、报告和历史仍然可以工作。因此它属于 B 类普通应用，而不是 Coding Agent Harness。

这种边界使核心结果可重复、可以离线测试，也不需要付费 API Key。它的限制同样明确：系统不能直接理解任意自然语言作业要求，也不会自动判断所有业务语义是否正确。只有当教师把要求写入规则或自动化测试时，系统才能确定性地发现不符合要求的行为。我认为这种诚实的能力边界比为了突出“AI”而加入自主循环更符合本项目的安全和可验证目标。

## 四、安全、分发与人的责任

安全方面最容易被夸大的是命令执行。参数数组、`shell=False`、环境白名单、超时和输出截断可以减少风险，但不能把本地 subprocess 变成完整沙箱。因此系统默认关闭不可信代码执行，公网 Render 环境也保持 `SUBMITREADY_ALLOW_UNTRUSTED_EXECUTION=false`。Docker 的非 root、只读文件系统和资源限制进一步降低风险，但应用内按任务创建无网隔离容器的 DockerRunner 仍然只是接口，不能声称已经实现完整沙箱。

凭据要求推动了范围收敛。本版本没有真实付费 LLM Provider，因此不收集 API Key，也避免用普通 `.env` 文件冒充安全存储。测试只使用明确的虚假凭据，GitHub 提交前执行凭据扫描。项目最终公开在 GitHub，远程 CI 成功，Docker 镜像由 GitHub Actions 和 Render 构建，并部署到 `https://submitready.onrender.com`。免费实例会休眠，SQLite 数据在重启后可能丢失，这些限制也必须在 README 中说明。

如果重新实现，我会更早建立共享 OpenAPI 类型或契约测试，再并行开发前后端；也会在第一个功能完成后立即用干净目录安装依赖，而不是等到最后。我还会在开发开始前确认课程指定技能、worktree 和不同类型智能体是否真正可用，避免事后只能记录偏差。AI 能显著提高实现和文档整理速度，但“项目是否真的解决问题”“证据是否足以支持完成声明”以及“哪些局限必须公开”，仍然需要由我本人判断和负责。
