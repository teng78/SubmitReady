# REFLECTION（AI 协助草稿，学生必须亲自审阅与改写）

> 学术规范提示：课程明确要求反思报告由学生本人撰写。本文件记录真实工程事实并提供批判性草稿，不能原样冒充学生的个人经历。提交前，学生必须逐段核对、补入自己的判断与实际操作，删除不符合本人经历的内容，并确认最终 1500–2500 字版本。

## Codex 的贡献与局限

Codex 对本项目最直接的贡献，是把一份很长的课程要求转成可验证的模块边界，并持续把“文件存在”改造成“命令实际通过”。它完成了后端、前端、样例、CI、Docker 配置和文档的初始实现，也适合处理机械但容易遗漏的工作，例如把 Zip Slip、盘符路径、UNC、符号链接、重复路径、加密成员和解压体积上限整理成同一套拒绝策略。并行任务把后端、前端和基础设施分开后，产出速度明显提高；目录所有权也减少了直接冲突。

但 Codex 生成的模块并不会天然互相兼容。最典型的错误是：后端 `/api/rules` 返回 `{rule:{...},revision,builtin}`，报告返回 `rule_name` 和 `return_code`；前端的 Mock 却假设规则已经扁平化、报告含 `rule` 和 `exit_code`。两边单独测试都通过，真实上传流程仍会崩溃。这说明“测试很多”不等于“测试到了系统边界”，Mock 如果由实现者凭想象编写，可能把同一个错误假设重复两遍。修复方式不是继续添加孤立单测，而是让前端测试使用真实 wire shape，再用浏览器走完上传—报告—历史路径。

第二个错误来自冷启动。主工作区已有前端依赖，所以 lint、类型检查和构建均通过；全新副本安装时，pnpm 11 却拒绝执行 esbuild 安装脚本。第一次冷启动因此失败。最终只允许 `esbuild: true` 后重新导出全新副本，才得到可信的通过结果。这个案例说明分发问题往往不在业务代码，而在锁文件、包管理器策略和隐式缓存。若没有干净副本，README 很容易写出“理论可用”的命令。

## 规约、计划与 TDD

独立上下文冷读是本次规约阶段最有价值的动作。冷读者没有先前对话，只看 SPEC/PLAN，就指出 YAML 字段、默认值、命令白名单、规则权威源和 ZIP 跨平台路径语义不完整。修订后，SafeExtractor 才有稳定错误码和清理职责，规则层才有精确结构与重复导入语义。这个反馈证明，规格不是为了展示完整，而是为了让陌生实现者在关键处不需要猜测。

当前环境没有课程指定的 Superpowers `test-driven-development` 技能，所以项目不能声称完整遵循了技能强制的红—绿—重构证据。后端任务保留了首次 `ModuleNotFoundError: app` 的红测和最终 21 tests green；前端保留了初次 lint 问题与最终门禁，但一些实现和测试是在同一子任务中快速形成，过程粒度不等同于严格的逐测试 TDD。这是需要学生诚实说明的偏差。自动化测试仍然起到了约束作用：API 适配修复后必须重新通过 ESLint、tsc、Vitest 和 build；pnpm 策略修复后必须从第二个全新副本重跑，而不是只改文档。

## 应用项目与 Harness 项目的区别

SubmitReady 的检查顺序、命令来源和停机条件都由普通代码固定。用户点击后，系统不会让 LLM 决定下一步、选择工具或根据结果修改代码。模板解释 Provider 只接收已产生、已截断和脱敏的摘要，单次输入、单次输出。即使完全删除解释模块，安全解压、规则、检查、Runner、报告和历史仍能工作。因此它是 B 类应用，而不是 Coding Agent Harness。

这种边界也有工程收益：核心结果可重复、离线可测、无需付费 Key，安全模型更容易审计。缺点是系统不会自动理解任意课程要求，也不会自行修复作业；这些是刻意的非目标，而不是“缺少 AI”。如果未来加入真实 LLM，首先要解决的是 keyring/secret 生命周期、数据最小化和供应商风险，而不是添加 Agent loop。

## 安全、分发与人的责任

安全方面最容易被夸大的部分是命令执行。参数数组、无 shell、环境白名单、超时和输出截断只能减少风险，不能把本机 subprocess 变成沙箱；Windows 网络阻断和完整子进程树终止也没有被证明。因此默认禁用执行是正确的失败关闭策略。Compose 的非 root、只读文件系统和资源限制进一步降低风险，但应用内 DockerRunner 仍只是接口，不能声称每次运行已有独立无网容器。

凭据要求反而推动了范围收敛：本版本没有真实付费 Provider，因而不收集 Key，也避免用 `.env` 明文冒充安全存储。如果以后接入供应商，必须先实现系统钥匙串或容器 secret 的录入、更新、清除和状态显示。敏感扫描测试中的凭据必须始终是明确的假值。

最终仍需人工验证的内容包括：在有 Docker 的机器完成构建与启动；检查远程 CI 最后一次状态；发布镜像与公网地址；用教师环境复核课程指定 Superpowers、worktree 和 PR 证据；逐段重写本反思。AI 可以生成大量工程产物，但“做对了吗”和“哪些证据足以支持声明”仍是人的责任。如果重做，我会先建立共享 OpenAPI 生成类型或契约测试，再并行前后端；会更早在干净目录安装依赖；也会在开发前确认课程强制技能是否真正可用，避免事后只能记录流程偏差。
